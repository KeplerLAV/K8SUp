# Déploiement du projet avec Helm et Kubernetes

Ce projet utilise Helm pour déployer l'ensemble de l'architecture Kubernetes (APIs, bases de données, composants web, messagerie et monitoring).

Toutes les commandes Helm doivent être lancées impérativement depuis le dossier `Helm/`.

---

## Prerequisites

### Outils obligatoires sur la machine

* Git
* Kubectl
* Helm v3+
* Un cluster Kubernetes actif (Docker Desktop, Minikube, AKS, etc.)

### Vérification rapide de l'environnement

```bash
kubectl version --client
helm version
kubectl get nodes

```

Si la commande `kubectl get nodes` renvoie vos nœuds actifs, le cluster est prêt à l'emploi.

## Déploiement pas à pas

### 1. Vérifier le chart Helm

```bash
helm lint .

```

*Résultat attendu :* `1 chart(s) linted, 0 chart(s) failed`. Si cette étape échoue, ne pas passer à la suite.

### 2. Générer les manifests (Test local recommandé)

```bash
helm template k8s . -n projet-apps > rendered.yaml

```

Cette commande génère l'intégralité des fichiers YAML Kubernetes sans rien appliquer au cluster, permettant de valider la syntaxe et les liaisons de variables au préalable.

### 3. Vérifications rapides du template

```powershell
Select-String -Path rendered.yaml -Pattern "<no value>"

```

S'il n'y a aucun résultat, cela signifie que tous les templates ont été correctement interprétés par Helm.

### 4. Déployer sur le cluster Kubernetes

```bash
helm upgrade --install k8s . \
  --namespace projet-apps \
  --create-namespace

```

| Paramètre | Rôle |
| --- | --- |
| `k8s` | Nom de la release Helm |
| `.` | Spécifie le chart courant |
| `--namespace projet-apps` | Namespace applicatif cible |
| `--create-namespace` | Force la création du namespace s'il n'existe pas |

## Vérifications après déploiement

### Visualisation des pods par namespace

```bash
kubectl get pods -n projet-apps
kubectl get pods -n ns-data
kubectl get pods -n ns-monitoring

```

*Note : Tous les pods doivent afficher le statut `Running`.*

### Visualisation des autres ressources

```bash
# Voir l'ensemble des services de tous les namespaces
kubectl get svc -A

# Voir les déploiements du namespace applicatif
kubectl get deploy -n projet-apps

```

## Accès à l'application Web

Si l'Ingress est activé dans votre fichier `values.yaml`, récupérez son point d'entrée :

```bash
kubectl get ingress -n projet-apps

```

Vous pouvez ensuite ouvrir votre navigateur et accéder à l'adresse suivante :
`http://localhost` (ou l'hôte spécifique configuré dans le fichier `values.yaml`).

## Maintenance du cycle de vie

### Mettre à jour le projet

Après toute modification du fichier `values.yaml` ou d'un fichier inclus dans le dossier `templates/`, réappliquez la configuration :

```bash
helm upgrade k8s . -n projet-apps

```

Kubernetes prendra en charge les modifications en mettant à jour uniquement les ressources impactées, sans coupure totale de l'infrastructure.

### Supprimer le projet

Pour désinstaller la release Helm :

```bash
helm uninstall k8s -n projet-apps

```

Pour effectuer un nettoyage complet et définitif des namespaces :

```bash
kubectl delete namespace projet-apps
kubectl delete namespace ns-data
kubectl delete namespace ns-monitoring

```

## Règles de gestion fondamentales

### À faire

* Effectuer toutes les modifications de configuration au sein du fichier `values.yaml`.
* Exécuter systématiquement les commandes Helm depuis le répertoire `Helm/`.
* Valider la conformité du chart à l'aide de `helm lint` avant chaque déploiement majeur.

### À ne pas faire

* Ne jamais modifier le fichier généré `rendered.yaml`.
* Ne pas modifier manuellement les manifests YAML générés à la main dans le cluster.
* Ne jamais appeler Helm depuis la racine du dépôt Git.
